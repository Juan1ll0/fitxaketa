# Plan de Implementación — 003.4 Tab Estadísticas

## Resumen

Implementar el tab `/estadisticas` con selector de periodo (Semana/Mes/Año), gráfica de barras Chart.js mostrando horas decimales por jornada, y resumen con métricas clave. Chart.js se instala en esta sub-spec y se carga vía lazy import para respetar el bundle budget.

## Dependencias

- **003.1** (Bottom Nav) — ya implementado, proporciona navegación al tab
- **db.ts** — ya exporta `getAllJornadas()` y `type Jornada`
- **Chart.js** — NO instalado, se instala en Fase 4

## Criterios de Aceptación (extraídos de la spec)

### Selector de periodo
- Botones "Semana" / "Mes" / "Año" visibles
- Default: "Mes" seleccionado
- Botón activo resaltado con color del theme
- Cambio reactivo de gráfica y resumen

### Gráfica de barras
- Chart.js con eje X (fechas DD/MM) y eje Y (horas decimales)
- Múltiples barras por día (una por jornada)
- Responsiva (se adapta al ancho)
- Colores del theme (`--color-primary`)
- Tooltip: fecha, hora inicio/fin, duración

### Resumen
- Total horas del periodo
- Media diaria (total / días trabajados)
- Días trabajados (días con ≥1 jornada)
- Número total de jornadas
- Formato: "40h 30m | 8h 6m/día | 5 días | 8 jornadas"

### Datos
- Fuente: `getAllJornadas()` de Dexie
- Filtrado por periodo con `filtrarPorPeriodo()`
- Solo jornadas cerradas (`status === 'closed'` y `duration !== null`)

### Rendimiento
- Chart.js lazy import (code-splitting)
- `export const ssr = false`
- Bundle budget: 150 KB entry / 300 KB total gzip

## Riesgos Técnicos

### 1. Bundle budget
- **Riesgo:** Chart.js (~70 KB gzip) puede acercar el bundle al límite de 300 KB total
- **Mitigación:** Lazy import con `import('chart.js')` para code-splitting. Verificar con `npm run size` después de implementar. Si se excede el budget, considerar:
  - Importar solo módulos necesarios (ya está en el plan)
  - Evaluar alternativas más ligeras (fuera de scope para esta spec)

### 2. Colores del theme en Chart.js
- **Riesgo:** Chart.js no lee CSS variables directamente
- **Mitigación:** Opciones:
  - Hardcodear colores con fallback (rgba basado en `--color-primary`)
  - Leer computed style en runtime (más complejo)
  - La spec sugiere usar `rgba(59, 130, 246, 0.7)` como ejemplo

### 3. Múltiples barras por día
- **Riesgo:** Si hay múltiples jornadas en un día, la gráfica puede volverse confusa
- **Mitigación:** La spec acepta múltiples barras por día (una por jornada). Labels del eje X muestran "DD/MM", por lo que habrá labels repetidos si hay múltiples jornadas el mismo día. Esto es comportamiento esperado.

### 4. SSR y Chart.js
- **Riesgo:** Chart.js requiere DOM, rompe en SSR
- **Mitigación:** `export const ssr = false` en `+page.svelte` (obligatorio). Ya contemplado en la spec.

### 5. Reactividad de Chart.js
- **Riesgo:** Chart.js no es reactivo por defecto; actualizar datos requiere `chart.update()`
- **Mitigación:** Usar `$effect` para detectar cambios en `jornadas` y llamar `chart.update()`. Destruir y recrear la instancia si es necesario (evaluar en implementación).

### 6. Zona horaria en filtrado de fechas
- **Riesgo:** `filtrarPorPeriodo` usa `new Date()` que puede tener comportamientos inconsistentes con zonas horarias
- **Mitigación:** Usar `setHours(0, 0, 0, 0)` para normalizar a medianoche local. Aceptar que el filtrado es en zona horaria del dispositivo (comportamiento esperado para app local).
