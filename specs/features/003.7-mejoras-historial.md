# 003.7 - Mejoras del Historial: Título, Filtros y Vista por Día

**Status:** done
**Creada:** 2026-06-23
**Autor:** sistema
**Depende de:** [003.3](./003.3-tab-historial.md), [003.6](./003.6-paginacion-estadisticas.md)
**Spec padre:** [003](./003-dashboard-componentes.md)

## Resumen

Mejorar la página de Historial (`/historial`) para convertirla en una vista de **consulta de solo lectura** con título propio, filtros temporales (reutilizando la navegación de periodos de la spec 003.6), filtro por fecha concreta, filtro por rango personalizado y filtro por estado. Cada día se muestra como un grupo colapsable (colapsado por defecto) con un resumen de horas, y cada jornada muestra hora de entrada/salida, duración redondeada y estado, sin el icono de sincronización. Se añade un botón "Exportar" que abre el flujo de exportación, dejando la generación real del fichero para una spec posterior.

## Contexto

La spec 003.3 implementó el Historial como un listado plano agrupado por día que muestra **todas** las jornadas sin filtros, con cabecera de página inexistente e icono de sync (✅/❌) que ya no aporta valor porque la sincronización no está implementada (spec 004).

La spec 003.6 introdujo `PeriodoNavegacion` + `filtrarPorPeriodo` para el tab Estadísticas. Esta spec reutiliza esa misma infraestructura en el Historial para que ambas vistas compartan modelo mental y código.

El **alta manual de fichajes** se decidió ubicar en la página principal (Fichar), no en el Historial, manteniendo el Historial como vista de solo lectura. Por tanto el alta manual queda **fuera de scope** de esta spec (ver [Fuera de scope](#fuera-de-scope)).

## Decisiones de UX (cerradas)

- **Solo lectura:** el Historial no edita ni crea jornadas. El alta manual vivirá en la página principal en una spec aparte.
- **Días colapsables, colapsados por defecto:** cada día muestra su resumen (total de horas) y se expande bajo demanda. Apropiado para periodos largos (mes/año).
- **Exportar — solo botón + flujo:** se añade el botón y la UI de exportación; la generación del fichero y el guardado en disco se dejan para una spec futura.
- **Filtros:** periodos predefinidos (semana/mes/año) + fecha concreta + rango personalizado (desde–hasta) + filtro por estado (todas/abiertas/cerradas).

## Historias de usuario

- Como usuario, quiero ver un título "Historial" en la página para saber dónde estoy, igual que en las demás páginas.
- Como usuario, quiero filtrar mis fichajes por semana, mes o año (igual que en Estadísticas) para acotar lo que veo.
- Como usuario, quiero saltar a una fecha concreta para ver solo los fichajes de ese día.
- Como usuario, quiero definir un rango personalizado (desde–hasta) para consultar un periodo arbitrario.
- Como usuario, quiero filtrar por estado (todas / abiertas / cerradas) para localizar jornadas sin cerrar.
- Como usuario, quiero ver por cada día la hora de entrada/salida real, la duración redondeada y el estado de cada fichaje.
- Como usuario, no quiero ver el icono de sincronización porque es secundario y no está implementado.
- Como usuario, quiero colapsar/expandir los fichajes de cada día para revisar periodos largos cómodamente, viendo el total del día cuando está colapsado.
- Como usuario, quiero un botón para exportar los fichajes del periodo filtrado.

## Criterios de aceptación

- [ ] AC-02: El `<title>` del documento sigue siendo "Historial - Fitxaketa".

- [ ] AC-07: Hay un selector de fecha (`<input type="date">`) que, al elegir un día, muestra únicamente los fichajes de ese día.
- [ ] AC-08: No permite seleccionar fechas futuras (atributo `max` = hoy).
- [ ] AC-09: Activar la fecha concreta tiene prioridad sobre el periodo y el rango; limpiarla devuelve al filtro de periodo activo.

### Filtro por rango personalizado

- [ ] AC-10: Hay dos selectores de fecha "Desde" y "Hasta" para definir un rango arbitrario (ambos inclusive, a nivel de día).
- [ ] AC-11: "Hasta" no permite fechas futuras; "Desde" no puede ser posterior a "Hasta" (se valida o se autocorrige).
- [ ] AC-12: Con un rango activo, el listado muestra solo las jornadas cuyo `start_time` cae dentro del rango (inclusive).
- [ ] AC-13: Un control claro permite volver al filtro de periodo (limpiar el rango).

### Filtro por estado

- [ ] AC-14: Conmutador con tres opciones: "Todas" (default), "Abiertas", "Cerradas".
- [ ] AC-15: El filtro por estado se combina (AND) con el filtro temporal activo (periodo, fecha concreta o rango).

### Vista por día (grupo colapsable)

- [ ] AC-16: Las jornadas se agrupan por día con cabecera "Hoy"/"Ayer"/"DD MMM YYYY" (reutiliza `agruparPorDia`).
- [ ] AC-17: Cada grupo de día está **colapsado por defecto**; al pulsar la cabecera se expande/colapsa.
- [ ] AC-18: La cabecera del día muestra siempre un **resumen** con el total de horas del día (duración efectiva/redondeada sumada), visible esté colapsado o expandido.
- [ ] AC-19: El estado de expansión es accesible (cabecera como `<button>` con `aria-expanded`).

### Cada jornada (fichaje)

- [ ] AC-20: Muestra hora de entrada real (HH:MM) y hora de salida real (HH:MM) o "En curso" si `end_time` es null.
- [ ] AC-21: Muestra la **duración redondeada** (efectiva) según la configuración de redondeo vigente, usando `duracionEfectivaMinutos`; "En curso" si la jornada está abierta.
- [ ] AC-22: Muestra el estado "Abierto"/"Cerrado" (o equivalente visual claro).
- [ ] AC-23: **No** se muestra el icono de sincronización (✅/❌).

### Botón Exportar (solo flujo/UI)

- [ ] AC-24: Hay un botón "Exportar" visible en la página de Historial.
- [ ] AC-25: Al pulsarlo se exporta el conjunto de jornadas **actualmente filtrado** (mismo criterio temporal + estado que el listado visible).
- [ ] AC-26: La generación del fichero y el guardado en disco son un placeholder explícito (no-op o aviso "Próximamente") — la implementación real es de otra spec.

### Estado vacío

- [ ] AC-27: Si el filtro activo no devuelve jornadas, se muestra un mensaje "No hay fichajes para este filtro" (en vez del "Aún no hay fichajes registrados", que se reserva para BD vacía).

### Modelo de filtros

Un único estado de "modo de filtro temporal" evita combinaciones ambiguas:

```typescript
type FiltroTemporal =
	| { tipo: 'periodo'; periodo: Periodo; fechaReferencia: Date }
	| { tipo: 'fecha'; fecha: Date }
	| { tipo: 'rango'; desde: Date; hasta: Date };

type FiltroEstado = 'todas' | 'abiertas' | 'cerradas';
```

- El filtro **temporal** es exclusivo (periodo / fecha / rango): activar uno desactiva los otros.
- El filtro de **estado** es ortogonal y se aplica en AND sobre el resultado temporal.
- Pipeline: `jornadas → (temporal) → (estado) → agruparPorDia → render`.

### Funciones nuevas (`historial-filtros.ts`)

```typescript
export function filtrarPorFecha(jornadas: Jornada[], fecha: Date): Jornada[];
export function filtrarPorRango(jornadas: Jornada[], desde: Date, hasta: Date): Jornada[];
export function filtrarPorEstado(jornadas: Jornada[], estado: FiltroEstado): Jornada[];
```

- Comparaciones a nivel de **día** (normalizar a 00:00 inicio / fin de día) usando los helpers de `fecha-negocio` cuando aplique.
- `filtrarPorRango` inclusivo en ambos extremos.

### DiaGroup colapsable

- Cabecera como `<button>` con `aria-expanded` y `aria-controls`.
- Resumen del día: suma de `duracionEfectivaMinutos` de las jornadas cerradas del día, formateada (ej. "8,2h"). Las jornadas abiertas se indican aparte o se omiten del total.
- Estado de colapsado local al componente, inicial `false` (colapsado).

### JornadaCard

- Eliminar el bloque del icono de sync (`{jornada.synced ? '✅' : '❌'}`) y su `aria-label`.
- Duración: `formatearDuracion(duracionEfectivaMinutos(jornada, redondeoMinutos))` en vez de `jornada.duration` directo.
- Añadir badge/etiqueta de estado "Abierto"/"Cerrado".

### Exportar (placeholder)

- Botón que recopila las `jornadasFiltradas` y llama a un `exportarJornadas(jornadas)` que por ahora muestra un aviso "Próximamente" (toast o similar) o es un no-op documentado.
- La firma se diseña pensando en la spec futura (formato, File System Access API / descarga).

## Fuera de scope

- **Alta manual de fichajes** (botón en la página principal que pide fecha + horas): es una feature de la página Fichar, no del Historial → spec aparte (p. ej. 003.8). El Historial permanece de solo lectura.
- **Edición/eliminación** de jornadas existentes.
- **Generación y guardado del fichero de exportación** (formato CSV/JSON, selección de carpeta, File System Access API) → spec aparte.
- **Sincronización con backend** y el icono de sync (spec 004).
- Búsqueda por texto libre / etiquetas.
