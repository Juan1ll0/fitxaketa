## Fase 2 — Lógica de negocio (utils puras)

### T2.1 — Utils de navegación temporal · `impl-svelte`

**Ficheros:** `src/lib/utils/dashboard-periodo.ts`.

**Modificar `filtrarPorPeriodo`** para aceptar `fechaRef` opcional (por defecto `new Date()`):

```typescript
export function filtrarPorPeriodo(
	jornadas: Jornada[],
	periodo: Periodo,
	primerDia: number = PRIMER_DIA_SEMANA,
	fechaRef?: Date // NUEVO: fecha de referencia (por defecto hoy)
): Jornada[] {
	const hoy = inicioDia(fechaRef ?? new Date(Date.now()));
	const { inicio, fin } = obtenerRangoPeriodo(periodo, hoy, primerDia);
	// ... resto igual
}
```

> **Cambio de firma compatible:** el nuevo parámetro es opcional y va al final. Los callers existentes (`+page.svelte` actual) no se rompen porque no lo pasan. Pero la página de estadísticas **debe** empezar a pasar `fechaReferencia` para que la paginación funcione (se hace en T3.2).

**Nuevas funciones:**

1. **`navegarPeriodo(periodo, fechaRef, direccion, primerDia)`** → `Date`
   - Crea copia de `fechaRef` (nunca muta).
   - `semana`: `±7` días (`setDate`).
   - `mes`: setear día a `1` primero (evitar overflow 31→feb), luego `setMonth(±1)`.
   - `trimestre`: `setMonth(±3)`.
   - `año`: `setFullYear(±1)`.
   - Devuelve la nueva fecha de referencia.

2. **`esPeriodoActual(periodo, fechaRef, hoy?, primerDia)`** → `boolean`
   - `hoy` por defecto `new Date()`.
   - Compara `obtenerRangoPeriodo(periodo, fechaRef)` con `obtenerRangoPeriodo(periodo, hoy)`: mismo `inicio.getTime()` = mismo periodo.
   - Esto es más robusto que comparar año/mes directamente (maneja semanas que cruzan años).

3. **`obtenerPuntoMedioPeriodo(periodo, fechaRef, primerDia)`** → `Date`
   - `semana`: día 4 de la semana (jueves, índice 3 desde el lunes). No se usa en la práctica (corto→largo no necesita punto medio), pero se implementa por completitud.
   - `mes`: día 15 del mes.
   - `trimestre`: mes central del trimestre, día 15.
   - `año`: 1 de julio del año.

**ACs cubiertos:** AC-04, AC-05, AC-06, AC-07 (navegación anterior/siguiente por periodos naturales), AC-22, AC-23 (coherencia entre vistas vía punto medio).
**Dependencias:** T1.1. **Paralelizable con:** T2.2 (ficheros distintos).

### T2.2 — Formateador de indicador de periodo · `impl-svelte`

**Ficheros:** `src/lib/utils/dashboard-format.ts`.

**Nueva función:**

```typescript
export function formatearIndicadorPeriodo(
	periodo: Periodo,
	fechaRef: Date,
	primerDia: number = 1

```

- **Semana:** `"Semana del 10 al 16 jun"` — usa `inicioSemana(fechaRef, primerDia)` para obtener el lunes, suma 6 días para el domingo. Formato: `día + mes corto` para ambos extremos. Si cruza meses: `"Semana del 30 jun al 6 jul"`. Si cruza años: `"Semana del 29 dic al 4 ene 2027"`.
- **Mes:** `"Junio 2026"` — nombre completo del mes + año. Capitalizado (ya hay patrón en `formatearFechaLarga`).
- **Trimestre:** `"Trimestre 2 2026"` — por completitud.
- **Año:** `"2026"` — solo el año.

> Usar `Intl.DateTimeFormat('es-ES', { month: 'long' })` para nombres de meses (consistente con el resto del código). Para el formato corto de semana, usar `{ day: 'numeric', month: 'short' }`.

   - Al cambiar de periodo largo→corto (Año→Mes, Mes→Semana), se llama `obtenerPuntoMedioPeriodo` para situar la referencia en el punto medio.
   - Esta lógica vive en el handler `onclick` del selector de periodo dentro del componente.

**Props (Svelte 5 runes):**

```typescript
let {
	periodo = $bindable(),
	fechaReferencia = $bindable(),
	primerDia = 1
}: {
	periodo: Periodo;
	fechaReferencia: Date;
	primerDia?: number;
} = $props();
```

> Usar `$bindable()` para `periodo` y `fechaReferencia` permite que la página padre use `bind:periodo` y `bind:fechaReferencia`, manteniendo el estado en la página (que es donde se necesita para `$derived` de filtrado).

**Estado interno:**

```typescript
let esActual = $derived(esPeriodoActual(periodo, fechaReferencia, new Date(), primerDia));
let indicador = $derived(formatearIndicadorPeriodo(periodo, fechaReferencia, primerDia));
```

**Handlers:**

```typescript
function anterior() {
	fechaReferencia = navegarPeriodo(periodo, fechaReferencia, 'anterior', primerDia);
}

function siguiente() {
	fechaReferencia = navegarPeriodo(periodo, fechaReferencia, 'siguiente', primerDia);
}

function irAHoy() {
	fechaReferencia = new Date();
}

function cambiarPeriodo(nuevo: Periodo) {
	const orden: Periodo[] = ['semana', 'mes', 'trimestre', 'año'];
	const idxActual = orden.indexOf(periodo);
	const idxNuevo = orden.indexOf(nuevo);

	if (idxNuevo > idxActual) {
		// Corto→largo: mantener referencia (ya cae en el periodo correcto)
		// No cambiar fechaReferencia
	} else if (idxNuevo < idxActual) {
		// Largo→corto: punto medio
		fechaReferencia = obtenerPuntoMedioPeriodo(periodo, fechaReferencia, primerDia);
	}

	periodo = nuevo;
}
```

**A11y:**

- `<nav aria-label="Navegación temporal">` envolviendo la barra.
- Botones con `aria-label` descriptivo: `"Periodo anterior"`, `"Periodo siguiente"`, `"Volver al periodo actual"`.
- Indicador de periodo con `aria-live="polite"` para que screen readers anuncien el cambio.
- Botones deshabilitados con `disabled` + `aria-disabled="true"` + `opacity-40 cursor-not-allowed`.

## Fase 4 — UI / decisiones UX pendientes

### T4.1 — Resolver decisiones UX y refinar estilos de navegación · `impl-ui`

**Ficheros:** `src/lib/components/PeriodoNavegacion.svelte`.

**Decisiones UX a resolver (de la spec § Decisiones UX):**

#### 1. Ubicación de los botones de navegación

Evaluar y decidir entre:

- **Opción A:** Barra encima de la gráfica (fuera del card)
- **Opción B:** Botones dentro del card de la gráfica (encima del canvas)
- **Opción C:** Botones en el pie del gráfico

**Criterios:** mobile-first, coherencia con el selector de periodo existente, no tapar la gráfica.

**Recomendación del planner:** Opción A — la barra de navegación temporal (selector + anterior/siguiente + hoy + indicador) como una unidad cohesiva encima de la gráfica. Es el patrón de Google Calendar y Apple Calendar. El indicador de periodo actúa como "título" de la barra.

#### 2. Indicador de periodo (formato y ubicación)

Evaluar y decidir entre:

- **Opción A:** Mantener indicador debajo de la gráfica
- **Opción B:** Mover indicador a la barra de navegación
- **Opción C:** Rango en botón central (`< 10-16 jun >`)

**Recomendación del planner:** Opción B — el indicador en la barra de navegación, entre los botones anterior/siguiente. Es el patrón estándar de calendarios. El indicador es clickeable (AC-14) para volver a hoy.

#### 3. Formato del indicador

Según la spec:

- **Semana:** `"Semana del 10 al 16 jun"` o `"10-16 jun 2026"`
- **Mes:** `"Junio 2026"` o `"Jun 2026"`
- **Año:** `"2026"`

**Recomendación del planner:** formato largo para semana (`"Semana del 10 al 16 jun"`), formato completo para mes (`"Junio 2026"`), solo año para año (`"2026"`). Si el espacio en móvil es insuficiente, evaluar formato corto.

**Estilos a refinar:**

- Botones anterior/siguiente: `rounded-full` o `rounded-lg`, tamaño táctil mínimo `44x44px` (WCAG 2.5.5).
- Botón "Hoy": `rounded-full px-3 py-1 text-xs` (compacto, estilo "chip").
- Botones deshabilitados: `opacity-40 cursor-not-allowed` (AC-21).
- Indicador de periodo: `font-semibold` cuando no es actual, `font-normal text-text-muted` cuando es actual.
- Transiciones suaves en cambios de periodo (`transition-opacity` en el indicador).
- Layout responsive: en móvil, la barra puede necesitar `flex-wrap` o layout en dos líneas (selector arriba, navegación abajo).

**ACs cubiertos:** refinamiento visual de AC-01, AC-02, AC-03, AC-08, AC-09, AC-13, AC-14, AC-21.
**Dependencias:** T3.1. **No paralelizable** con T3.1 (mismo fichero).

**ACs cubiertos:** AC-01, AC-02, AC-03, AC-09, AC-13, AC-14, AC-21, AC-22, AC-23.

**Dependencias:** las utils/store correspondientes. **Paralelizable:** sí (ficheros disjuntos).

## Riesgos técnicos

1. **Límite de tamaño de `dashboard-periodo.ts` (55 → ~110 líneas).** Añadir 4 funciones (~55 líneas) lleva el fichero cerca del límite de 120 líneas. _Mitigación:_ si supera 120, extraer las funciones de navegación a `dashboard-navegacion.ts` (nuevo, puro) y reexportar desde el barrel. Verificar con `wc -l` tras implementar T2.1.

2. **Overflow de días en navegación de meses.** `setMonth(mes - 1)` con día 31 puede desbordar a marzo (31 feb → 3 mar). _Mitigación:_ T2.1 setea día a `1` antes de cambiar el mes. Documentar este comportamiento en el código.

3. **Semanas que cruzan años/meses.** `formatearIndicadorPeriodo` debe manejar correctamente semanas como "29 dic 2025 al 4 ene 2026". _Mitigación:_ comparar año de inicio y fin; si difieren, mostrar año en ambos extremos. Test T5.2 cubre este edge case.

4. **`+page.svelte` tras extracción (139 → ~130 líneas).** La extracción de `PeriodoNavegacion` reduce la página, pero añadir `fechaReferencia` y el reset la mantiene cerca de 130-140 líneas. _Mitigación:_ verificar con `wc -l` tras T3.2. Si supera 150, extraer la sección de resumen a `ResumenEstadisticas.svelte`.

5. **`$bindable()` en Svelte 5.** El componente `PeriodoNavegacion` usa `$bindable()` para `periodo` y `fechaReferencia`. Esto requiere que la página padre use `bind:periodo` y `bind:fechaReferencia`. _Riesgo:_ si el implementer no está familiarizado con `$bindable()`, puede usar el patrón antiguo de callbacks. _Mitigación:_ el contrato en T1.1 documenta el uso de `$bindable()`. Verificar que `svelte-check` no reporta warnings.

6. **`new Date()` en `$derived` de `esPeriodoActual`.** El `$derived` que calcula `esActual` usa `new Date()` como `hoy`. Esto se evalúa una vez en el render. Si el usuario deja la app abierta durante la medianoche, el cálculo podría quedar desactualizado. _Mitigación:_ riesgo aceptable (edge case extremo). Si se quisiera cubrir, añadir un `setInterval` que actualice un `hoy` state cada minuto (patrón ya usado en `+page.svelte` del tab Fichar). No planificar a menos que el tester lo detecte.

7. **`afterNavigate` vs `onMount` para reset (AC-24).** `afterNavigate` se ejecuta en cada navegación a la página (incluyendo la primera). `onMount` solo en la primera. La spec dice "al entrar por primera vez al tab", pero en la práctica SvelteKit mantiene el componente montado si se navega entre tabs con `BottomNav`. _Mitigación:_ usar `afterNavigate` (ya existente en la página) que cubre ambos casos. Si el comportamiento deseado es solo la primera vez, cambiar a `onMount`. Decisión para @impl-ui durante implementación.

8. **knip (código muerto).** Nuevos exports del barrel (`navegarPeriodo`, `esPeriodoActual`, `obtenerPuntoMedioPeriodo`, `formatearIndicadorPeriodo`) deben tener consumidor. _Mitigación:_ el componente `PeriodoNavegacion` los consume. Verificar `npm run knip` al final.

9. **jscpd (duplicación ≤2%).** Los handlers `anterior`/`siguiente`/`irAHoy` son triviales y no deberían generar duplicación. _Mitigación:_ verificar `npm run dup` al final.

## Fuera de scope (de la spec, reiterado)

- Selector de fecha manual (calendario).
- Saltar directamente a un mes/año específico (dropdown).
- Comparación entre periodos.
- Exportación de datos del periodo seleccionado.
- Navegación de trimestre en la UI (el tipo lo incluye, las utils lo cubren, pero no hay botón).
- Cambios en el store global, Dexie, Chart.js o backend GAS.
