### T2.3 — Utils puras de redondeo · `impl-svelte`

**Ficheros:** `src/lib/utils/redondeo.ts` (NUEVO, puro).

- `redondearFecha(fecha: Date, minutos: number): Date` → redondeo al múltiplo **más cercano** del intervalo (no muta; `minutos===0` → copia sin cambio). Ej.: 21:43 con 15 → 21:45; 21:37 → 21:30.
- `duracionEfectivaMinutos(jornada: Jornada, snapshots: Settings[]): number` → lee `settingsVigente(snapshots, jornada.start_time)`, redondea `start_time` y/o `end_time` según `redondeo_aplicar_a`, devuelve `(endR − startR)/60000`. **No muta** la jornada. Si `end_time` es null → 0 (o duración hasta ahora, según convención de jornada cerrada; aplicar a jornadas cerradas).

> **B1 — Verificación obligatoria de pureza (depcruise).** Estas utils importan `type { Jornada }` y `type { Settings }` desde `$lib/db`. La regla `utils-son-puras` de `.dependency-cruiser.cjs` prohíbe que `^src/lib/utils` importe de `^src/lib/db`. **Hoy `type { Jornada } from '$lib/db'` NO rompe la regla** (depcruise no sigue los imports `type`-only de `db.ts`), así que se asume que `type { Settings }` tampoco. **No darlo por resuelto:** tras crear esta util ejecutar `npm run depcruise`. Si la regla se dispara con `import type { Settings } from '$lib/db'`, aplicar el **fallback**: crear un módulo puro `src/lib/db-types.ts` con las interfaces `Jornada` y `Settings`, reexportarlas desde `db.ts` (`export type { Jornada, Settings } from './db-types';`) y reapuntar los imports de tipo de las utils a `$lib/db-types`. Aplica igual a T2.4.

**ACs:** redondeo al más cercano, aplicado al vuelo sin mutar, snapshot vigente al `start_time`.
**Dependencias:** T1.1, T2.2 (`settingsVigente`). **Paralelizable con:** T2.1. **Verificación de cierre:** `npm run depcruise` verde (ver nota B1; fallback `db-types.ts` si falla).

## Riesgos técnicos

1. **Límites de tamaño (ficheros ≤120/150, funciones ≤50, complejidad ≤10).** `dashboard-grafica.ts` ya está en 128 líneas y `StatsChart.svelte` en 132; añadir objetivo/balance por label y el plugin inline desbordaría. _Mitigación (N1, **no condicional**):_ T2.5 crea **desde el inicio** el helper puro `dashboard-grafica-objetivo.ts` con el cálculo de objetivo/balance por label, y mantiene el plugin del chart conciso. Verificar líneas reales con `npm run check` tras implementar.
2. **`any` prohibido en plugin de Chart.js.** El contexto del plugin (`afterDraw`, escalas) tiende a tipos laxos. _Mitigación:_ tipar con los tipos de `chart.js` (`Chart`, `Plugin`) en lugar de `any`; replicar el tipado ya usado en `fitxaketaDataLabels`.
3. **Bundle budget (size-limit).** La línea de objetivo se hace con **plugin inline**, NO con `chartjs-plugin-annotation`. Cualquier dependencia nueva rompería el presupuesto. _Riesgo de regresión si un implementer recurre a la librería._ Declarado explícitamente como prohibido.
4. **Migración Dexie v4.** Solo añade tabla `settings`; las jornadas no se tocan (todo al vuelo). Riesgo: olvidar `seedSettingsIfEmpty` deja `settingsVigente()` sin snapshot. _Mitigación:_ seed en init y test T5.5; fallback "más antiguo" en `settingsVigente`.
5. **Cambio de firma con estado (B3).** `calcularResumenPeriodo`, `prepararDatosGrafica`, `obtenerRangoPeriodo` **y `filtrarPorPeriodo`** pasan de "sin estado" a recibir `snapshots`/`primerDia`. _Riesgo:_ callers desactualizados (página estadísticas, store, barrel `dashboard.ts`, tests existentes). _Mitigación:_ T2.5 actualiza el barrel y las firmas; T3.3 actualiza el caller `estadisticas/+page.svelte` (pasa `settingsActual(settings).primer_dia_semana` a `filtrarPorPeriodo`/`obtenerRangoPeriodo` y `settings` a resumen/gráfica); T5.4/T5.6/T5.7 actualizan tests en bloque; ejecutar `npm run check` tras la fase 2.
6. **dependency-cruiser (capas) — B1.** `settings.ts`, `redondeo.ts`, `dashboard-exceso.ts`, `dashboard-grafica-objetivo.ts` deben ser **puras**: sin importar componentes ni Dexie (`db.ts`) — solo tipos. La regla `utils-son-puras` prohíbe `^src/lib/utils → ^src/lib/db`. Hoy `type { Jornada } from '$lib/db'` **NO** rompe la regla (depcruise no sigue imports `type`-only de `db.ts`), por lo que se asume que `type { Settings }` tampoco. _Verificación obligatoria:_ ejecutar `npm run depcruise` tras crear cada util (T2.3, T2.4, T2.5). _Fallback si falla:_ crear módulo puro `src/lib/db-types.ts` con `Jornada` y `Settings`, reexportarlos desde `db.ts`, y apuntar los imports `type` de utils a `$lib/db-types`. La lógica con Dexie (helpers, seed, descarte) sigue en `db.ts`/store.
7. **knip (código muerto).** Nuevos exports (`addSettingsSnapshot`, `seedSettingsIfEmpty`, `getSettings`, etc.) deben tener consumidor o quedar como entry. Las rutas nuevas ya las cubre `src/routes/**/+*`. _Mitigación:_ no exportar helpers no usados; verificar `npm run knip` al final.
8. **jscpd (duplicación ≤2%).** El plugin de la línea y el tooltip pueden parecerse a `fitxaketaDataLabels`. _Mitigación:_ factorizar lo común si jscpd marca el bloque.
9. **a11y (svelte-check `--fail-on-warnings`).** Form de configuración y filas de acciones requieren `<label for>`, `aria-*`, foco; color no como único indicador en la caja Exceso (resuelto con el signo). _Mitigación:_ T3.1/T3.3/T4.3/T4.4 lo contemplan; `npm run check` lo valida.
