### T2.1 — `alta-manual.ts`: combinar y validar · `impl-svelte`

**Ficheros:** `src/lib/utils/alta-manual.ts` (NUEVO).

```typescript
// Combina una fecha (Date) + "HH:MM" en un Date LOCAL (evita el bug UTC de new Date("..."))
export function combinarFechaHora(fecha: Date, hora: string): Date {
	const [h, m] = hora.split(':').map(Number);
	const d = new Date(fecha);
	d.setHours(h, m, 0, 0);
	return d;
}

// null = OK; string = mensaje de error a mostrar.
export function validarAltaManual(
	start: Date,
	end: Date,
	minJornadaMinutos: number
): string | null {
	if (start.getTime() > Date.now()) return 'La fecha/hora de inicio no puede ser futura.';
	if (end.getTime() <= start.getTime()) return 'La hora de fin debe ser posterior a la de inicio.';
	const dur = (end.getTime() - start.getTime()) / 60000;
	if (minJornadaMinutos > 0 && dur < minJornadaMinutos)
		return `La jornada debe durar al menos ${minJornadaMinutos} min.`;
	return null;
}
```

> **Cruce de medianoche fuera de scope:** como `start`/`end` se construyen sobre la **misma fecha**, `end <= start` captura el caso (error). El soporte real de cruce es spec futura.
> **Bug de zona horaria:** construir con `setHours` sobre la fecha local, nunca `new Date("YYYY-MM-DDTHH:MM")`.

**ACs cubiertos:** AC-20, AC-21, AC-22 (lógica), AC-23 (deriva start/end).
**Dependencias:** ninguna. **Paralelizable con:** T2.2, T2.3.

## Riesgos técnicos

1. **Zona horaria en `combinarFechaHora`.** `new Date("YYYY-MM-DDTHH:MM")` interpreta UTC y desfasa. _Mitigación:_ construir con `setHours` sobre la fecha local; test de medianoche (T5.1).

2. **¿`resumen.totalHoras` aplica redondeo?** AC-10 exige duración efectiva/redondeada coherente con Historial/Estadísticas. _Mitigación:_ verificar `calcularHoy`; si no aplica `redondeo_minutos`, recalcular en `ResumenStatCards` con `duracionEfectivaMinutos` (T3.4).

3. **Reloj cada segundo vs rendimiento/limpieza.** Un `setInterval` adicional para `ahora` debe limpiarse en `onDestroy` y no duplicar el tick del cronómetro. _Mitigación:_ un único intervalo de 1s que actualice `ahora`; reutilizar el `subscribe` existente para el resto (T3.7).

4. **Scaffold sin sobre-ingeniería.** El registro de slots no debe convertirse en un sistema de plugins completo (fuera de scope). _Mitigación:_ registro **estático** + mapa local en la página; sin UI ni persistencia (AC-15). Mantener simple.

5. **Tamaño de `+page.svelte`.** Estado del reloj + mapa de slots + modal puede pasar de 150 líneas. _Mitigación:_ extraer `DashboardSlots.svelte` si hace falta; `wc -l` tras T3.7.

6. **dependency-cruiser.** `dashboard-layout.ts` (utils) **no** puede importar componentes; el mapa `variante→componente` vive en la página. `alta-manual.ts` solo `import type`. _Mitigación:_ verificar `npm run depcruise`.

7. **knip (código muerto).** Nuevos exports (`crearJornadaManual`, `agregarJornadaManual`, utils, `dashboardLayoutPorDefecto`, componentes) deben tener consumidor real. _Mitigación:_ la página y la modal los consumen; `npm run knip` al final.

8. **Accesibilidad de la modal.** Trampa/restitución de foco y cierre con `Esc` son fáciles de olvidar. _Mitigación:_ preferir `<dialog>` nativo (gestiona foco/Esc); cubrir en T4.3 y T5.3.

9. **Jerarquía visual (altitud).** El acceso al alta y el banner E4 no deben competir con el botón Fichar. _Mitigación:_ decisión cerrada (secundario) + verificación visual en T4.2/T4.4.
