# 003.6 - Paginación Temporal en Estadísticas

**Status:** done
**Creada:** 2026-06-22
**Autor:** sistema
**Depende de:** [003.4](./003.4-tab-estadisticas.md)
**Spec padre:** [003](./003-dashboard-componentes.md)

## Resumen

Añadir navegación temporal (anterior/siguiente) al tab Estadísticas, permitiendo al usuario moverse entre semanas, meses o años. Los totales y la gráfica se recalculan reactivamente según el periodo seleccionado. No se puede navegar más allá de la fecha actual.

## Contexto

La spec 003.4 implementó el selector de tipo de periodo (Semana/Mes/Año) pero siempre muestra el periodo actual. El usuario necesita poder navegar hacia atrás en el tiempo para consultar estadísticas históricas, manteniendo el límite de la fecha actual como frontera superior.

## Historias de usuario

- Como usuario, quiero navegar entre semanas (anterior/siguiente) para ver estadísticas de semanas pasadas
- Como usuario, quiero navegar entre meses (anterior/siguiente) para ver estadísticas de meses pasados
- Como usuario, quiero navegar entre años (anterior/siguiente) para ver estadísticas de años pasadas
- Como usuario, quiero que los totales (horas, media, días, jornadas) se recalculen automáticamente según el periodo que estoy visualizando
- Como usuario, no quiero poder navegar a periodos futuros (semanas/meses/años posteriores al actual)
- Como usuario, quiero poder volver rápidamente al periodo actual (hoy/esta semana/este mes/este año) sin tener que pulsar "Siguiente" múltiples veces

## Criterios de aceptación

### Navegación temporal

- [x] AC-01: Botones "Anterior" y "Siguiente" visibles junto al selector de periodo
- [x] AC-02: El botón "Siguiente" se deshabilita cuando el periodo mostrado es el actual (no se puede ir al futuro)
- [x] AC-03: El botón "Anterior" siempre está habilitado (no hay límite inferior)
- [x] AC-04: Al pulsar "Anterior" en vista Semana, se retrocede 7 días (semana natural anterior)
- [x] AC-05: Al pulsar "Anterior" en vista Mes, se retrocede 1 mes (mes natural anterior)
- [x] AC-06: Al pulsar "Anterior" en vista Año, se retrocede 1 año (año natural anterior)
- [x] AC-07: Al pulsar "Siguiente", se avanza al periodo siguiente (semana/mes/año)
- [x] AC-08: El indicador de periodo muestra el rango actual (ej: "Semana del 10 al 16 jun", "Junio 2026", "2026")

### Volver al periodo actual

- [x] AC-09: Botón "Hoy" (o similar) visible para volver rápidamente al periodo actual
- [x] AC-10: Al pulsar "Hoy" en vista Semana, se muestra la semana actual
- [x] AC-11: Al pulsar "Hoy" en vista Mes, se muestra el mes actual
- [x] AC-12: Al pulsar "Hoy" en vista Año, se muestra el año actual
- [x] AC-13: El botón "Hoy" se deshabilita cuando ya se está visualizando el periodo actual
- [x] AC-14: Alternativa UX: pulsar sobre el indicador de fecha/periodo también vuelve al periodo actual (evaluar viabilidad)

### Recálculo de datos

- [x] AC-15: Al cambiar de periodo (anterior/siguiente), la gráfica se actualiza con los datos del nuevo rango
- [x] AC-16: Al cambiar de periodo, el resumen (total horas, media diaria, días trabajados, jornadas) se recalcula para el nuevo rango
- [x] AC-17: Si no hay datos en el periodo seleccionado, se muestra "No hay datos para este periodo"

### Límite superior (fecha actual)

- [x] AC-18: En vista Semana, no se puede navegar a una semana posterior a la semana actual
- [x] AC-19: En vista Mes, no se puede navegar a un mes posterior al mes actual
- [x] AC-20: En vista Año, no se puede navegar a un año posterior al año actual
- [x] AC-21: El botón "Siguiente" se deshabilita visualmente (disabled + opacity reducida) al llegar al periodo actual

### Coherencia entre vistas (UX)

- [x] AC-22: Al cambiar de tipo de periodo corto→largo (Semana→Mes→Año), se mantiene la referencia temporal del periodo visualizado
- [x] AC-23: Al cambiar de tipo de periodo largo→corto (Año→Mes→Semana), se sitúa en el punto medio del periodo para minimizar navegación
- [x] AC-24: El estado de navegación se resetea al periodo actual al entrar por primera vez al tab

## Decisiones UX (para el subagente @impl-ui)

Las siguientes decisiones deben ser estudiadas y resueltas por el subagente de UX durante la planificación:

### 1. Ubicación de los botones de navegación

Evaluar las siguientes opciones y decidir la más coherente:

- **Opción A:** Barra de navegación encima de la gráfica (fuera del card de la gráfica)
- **Opción B:** Botones integrados dentro del card de la gráfica (encima del canvas)
- **Opción C:** Botones en el pie del gráfico (debajo del canvas)

Criterios de evaluación:

- Espacio disponible en móvil (la app es mobile-first)
- Coherencia visual con el selector de periodo existente (Semana/Mes/Año)
- No tapar la gráfica ni reducir su área útil excesivamente

### 2. Reglas de navegación entre vistas (DECIDIDO)

**Navegación de periodo corto a largo (Semana → Mes → Año):**

- Mantener la referencia temporal del periodo visualizado
- Ejemplo: Estoy viendo la semana 3 de Junio 2026 → pulso "Mes" → me sitúo en Junio 2026 → pulso "Año" → me sitúo en 2026

**Navegación de periodo largo a corto (Año → Mes → Semana):**

- Usar el punto medio del periodo para minimizar navegación posterior
- Ejemplo: Estoy viendo el año 2026 → pulso "Mes" → me sitúo en Junio/Julio 2026 (mes central) → pulso "Semana" → me sitúo en la semana central de ese mes
- Justificación: desde el punto medio, el usuario necesita como máximo N/2 pulsaciones para llegar a cualquier extremo del periodo

### 3. Tipo de navegación: ¿Libre o Jerárquica con contexto фиjado? (DECIDIDO)

**Decisión: Opción A - Navegación Libre**

**Justificación:**

1. **Mobile-first**: La app es mobile-first. La Opción B requiere breadcrumb visual que consume espacio en pantallas pequeñas sin aportar funcionalidad crítica.

2. **Patrones establecidos**: Google Calendar, Apple Calendar y apps de productividad temporal usan navegación libre. Los usuarios están familiarizados con este patrón.

3. **Caso de uso**: Para fichaje de horas, el usuario quiere consultar historial ("semana pasada", "mes anterior", "comparar Diciembre 2025 vs Enero 2026"). No necesita drill-down jerárquico. El tiempo es dimensión lineal, no estructura anidada.

4. **Simplicidad**: La Opción B añade estado adicional (`contextoFijado`), lógica de límites, breadcrumb navegable y reglas de liberación de contexto. Mayor complejidad sin beneficio claro.

5. **Flexibilidad**: Con navegación libre, el usuario salta directamente entre periodos adyacentes para comparar. Con Opción B, fricción innecesaria (subir a vista Año primero).

**Comportamiento definido:**

- El usuario navega libremente entre cualquier semana/mes/año sin restricciones de contexto
- Botones "Anterior/Siguiente" funcionan sin límites de contexto (solo límite superior: periodo actual)
- Ejemplo: Puede navegar de Diciembre 2025 a Enero 2026 directamente en vista Mes
- El indicador de periodo (AC-08) muestra claramente el periodo actual, proporcionando contexto suficiente
- No se implementa breadcrumb ni contexto fijado
- El botón "Hoy" (AC-09) permite volver al periodo actual desde cualquier punto

**Nota**: Las decisiones pendientes (ubicación de botones, indicador de periodo) se resolverán durante la implementación.

### 4. Indicador de periodo actual

Evaluar si el pie de la gráfica debe moverse a la barra de navegación:

- **Opción A:** Mantener el pie debajo de la gráfica (como está ahora)
- **Opción B:** Mover el indicador de periodo (ej: "Junio 2026") a la barra de navegación, junto a los botones anterior/siguiente
- **Opción C:** Mostrar el rango de fechas en el botón central (ej: "< 10-16 jun >" o "< Junio 2026 >")

Criterios:

- Claridad visual: el usuario debe saber en qué periodo está de un vistazo
- Espacio: no sobrecargar la interfaz
- Si se implementa "pulsar sobre fecha para volver a hoy" (AC-14), el indicador debe ser claramente clickeable

### 5. Formato del indicador de periodo

Según la vista:

- **Semana:** "Semana del 10 al 16 jun" o "10-16 jun 2026"
- **Mes:** "Junio 2026" o "Jun 2026"
- **Año:** "2026"

## Edge cases

- ¿Qué pasa si el usuario está en la semana actual y pulsa "Siguiente"? → Botón deshabilitado, no hace nada
- ¿Qué pasa si no hay datos en un periodo histórico? → Mensaje "No hay datos para este periodo"
- ¿Qué pasa si el usuario cambia de vista (Semana→Mes) y luego vuelve a la vista original? → Se mantiene la referencia temporal coherente (ver sección 3 de Decisiones UX)
- ¿Qué pasa con semanas que cruzan meses/años? → La semana se define por su lunes (inicio de semana), el indicador muestra el rango completo
- ¿Qué pasa con el cambio de hora (DST)? → No afecta, los fichajes se guardan con timestamp UTC
- ¿Qué pasa si pulso "Hoy" estando ya en el periodo actual? → Botón deshabilitado, no hace nada
- ¿Qué pasa si estoy en vista Año 2025 y pulso "Hoy"? → Salta al año actual (2026)

## Fuera de scope

- Selector de fecha manual (calendario)
- Saltar directamente a un mes/año específico (dropdown de meses/años)
- Comparación entre periodos (ej: este mes vs el mes pasado)
- Exportación de datos del periodo seleccionado
