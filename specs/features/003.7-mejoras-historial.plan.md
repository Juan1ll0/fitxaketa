## Fase 1 — Tipos TypeScript / contratos de datos

### T1.1 — Definir tipos de filtros y contratos · `impl-svelte`

**Ficheros:** `src/lib/utils/historial-filtros.ts` (NUEVO — declarar tipos aquí; las funciones se implementan en Fase 2).

**Tipos:**

```typescript
import type { Periodo } from '$lib/stores/app-state';

export type FiltroTemporal =
	| { tipo: 'periodo'; periodo: Periodo; fechaReferencia: Date }
	| { tipo: 'fecha'; fecha: Date }
	| { tipo: 'rango'; desde: Date; hasta: Date };

export type FiltroEstado = 'todas' | 'abiertas' | 'cerradas';
```

**Contratos de funciones puras (Fase 2):**

```typescript
export function filtrarPorFecha(jornadas: Jornada[], fecha: Date): Jornada[];
export function filtrarPorRango(jornadas: Jornada[], desde: Date, hasta: Date): Jornada[];
export function filtrarPorEstado(jornadas: Jornada[], estado: FiltroEstado): Jornada[];
```

**Contrato del componente `HistorialFiltros.svelte`:**

```typescript
let {
	filtroTemporal = $bindable<FiltroTemporal>(),
	filtroEstado = $bindable<FiltroEstado>(),
	primerDia = 1,
	onExportar
}: {
	filtroTemporal: FiltroTemporal;
	filtroEstado: FiltroEstado;
	primerDia?: number;
	onExportar: () => void;
} = $props();
```

> El filtro **temporal es exclusivo** (periodo / fecha / rango): se modela como unión discriminada, no como tres estados independientes, para evitar combinaciones ambiguas (AC-09, AC-13).

**ACs cubiertos:** soporte de contrato para AC-03→AC-15, AC-24.
**Dependencias:** ninguna. **Bloquea:** Fases 2-4.

### T2.1 — `historial-filtros.ts`: filtros temporales y de estado · `impl-svelte`

**Ficheros:** `src/lib/utils/historial-filtros.ts`.

1. **`filtrarPorFecha(jornadas, fecha)`** → jornadas cuyo `start_time` cae el mismo día que `fecha`.
   - Usar `mismoDia(jornada.start_time, fecha)` de `fecha-negocio`.

2. **`filtrarPorRango(jornadas, desde, hasta)`** → jornadas con `start_time` entre `inicioDia(desde)` y fin de `hasta` (inclusive ambos extremos, AC-12).
   - `const ini = inicioDia(desde); const fin = inicioDia(hasta); fin.setDate(fin.getDate() + 1);`
   - Filtro: `start_time >= ini && start_time < fin`.

3. **`filtrarPorEstado(jornadas, estado)`** → según `'todas'` (sin filtro), `'abiertas'` (`status === 'open'`), `'cerradas'` (`status === 'closed'`).

## Riesgos técnicos

1. **`duracionEfectivaMinutos` necesita los snapshots de settings, no minutos.** `JornadaCard` y `DiaGroup` deben recibir `snapshots: Settings[]` y propagarlos desde la página. _Mitigación:_ la página ya carga `settings` (patrón de estadísticas); pasarlos como prop. NO usar `jornada.duration` directo (no aplica redondeo histórico).

2. **Exclusividad de filtros temporales.** Modelar periodo/fecha/rango como tres estados independientes provoca combinaciones ambiguas. _Mitigación:_ unión discriminada `FiltroTemporal` (T1.1); el dispatcher `aplicarFiltroTemporal` garantiza un solo modo activo.

3. **Tamaño de `HistorialFiltros.svelte`.** Tres modos temporales + estado + exportar pueden superar 120 líneas. _Mitigación:_ extraer `RangoFechas.svelte` si hace falta. Verificar `wc -l` tras T3.3.

4. **Tamaño de `historial/+page.svelte`.** Pipeline + doble estado vacío + reset puede acercarse a 150. _Mitigación:_ mover `aplicarFiltroTemporal` a `historial-filtros.ts`; verificar `wc -l` tras T3.4.

5. **`<input type="date">` y zonas horarias.** El valor del input es `YYYY-MM-DD` en hora local; construir `new Date(valor)` interpreta UTC y puede desfasar un día. _Mitigación:_ parsear con `new Date(y, m-1, d)` o helper de `fecha-negocio`; cubrir con test de medianoche.

6. **knip (código muerto).** `exportarJornadas` y los nuevos exports de `historial-filtros.ts` deben tener consumidor. _Mitigación:_ el botón (T3.4) y la página consumen todo. Verificar `npm run knip` al final.

7. **jscpd (duplicación ≤2%).** `DiaGroup` y `JornadaCard` comparten poco; los filtros son triviales. _Mitigación:_ verificar `npm run dup` al final.

8. **dependency-cruiser.** `historial-filtros.ts` solo puede importar `type Periodo` (tipo, no runtime) y utils de `fecha-negocio`/`db` types. _Mitigación:_ usar `import type` para `Periodo` y `Jornada`/`Settings`. Verificar `npm run depcruise`.

9. **Reset en `afterNavigate` vs estado persistente.** Igual que en estadísticas (riesgo conocido 003.6): `afterNavigate` resetea filtros en cada entrada al tab. _Mitigación:_ comportamiento deseado (entrar al Historial siempre muestra el mes actual, todas). Si se quiere persistir el filtro entre navegaciones, decisión de @impl-ui.
